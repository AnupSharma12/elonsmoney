# Spend Elon's Money

A fun, interactive web app that lets you spend Elon Musk's $830 billion fortune on everything from candy to cruise ships. See how much you can spend and what tiny percentage of his wealth you've actually used!


## Try It Live

**[Open the app →](https://anupsharma12.github.io/elonsmoney)**

Or just open `index.html` in your browser - no build step needed.

## Quick Start

```bash
git clone https://github.com/AnupSharma12/elonsmoney.git
cd elonsmoney
# Open index.html in your browser
```

That's it. Pure HTML, CSS, and vanilla JavaScript. No dependencies, no bundler, no fuss.

## Features

- **$830 Billion Starting Balance** - Spend Elon Musk's entire net worth on a wide variety of items
- **40+ Items to Buy** - From $1 candy to $44 billion Twitter (X), with everything in between
- **Real-time Balance Updates** - Watch your balance decrease (or increase) as you buy and sell items
- **Smart Purchase Controls** - Can't buy what you can't afford; sell items to get money back
- **Detailed Receipt** - See exactly what you bought, total spent, and what percentage of Elon's fortune you used
- **Sticky Balance Bar** - Your current balance stays visible as you scroll through items
- **Responsive Grid Layout** - Clean, organized item cards with images and easy controls

## How to Run Locally

1. Clone or download this repository
2. Open `index.html` in any modern web browser (Chrome, Firefox, Safari, Edge)
3. Start spending! 💸

No server required. No `npm install`. No environment variables. It just works.

## How It Works

The app is built with zero external dependencies. All state is managed in a single `script.js` file using vanilla DOM APIs. Each item in the catalog is a DOM element with data attributes; on load, the script walks the DOM, builds an in-memory state array, and attaches event listeners. Buying and selling updates both the DOM and the shared balance in real time.

The receipt section at the bottom is dynamically generated by filtering the items array for quantities greater than zero, then rendering name × quantity and total cost. The percentage calculation divides total spent by the starting $830 billion fortune.

I chose this approach over a framework like React or Vue because the project is small enough that the overhead of a build step and virtual DOM would add complexity without improving the user experience. The entire app loads in under 100KB and works offline.
